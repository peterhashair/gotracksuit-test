// System design · Architecture · Scalability

Q — 01

When building an API endpoint, there are typically a few distinct layers involved. Can you briefly describe what each of those layers is responsible for and why it's useful to keep them separate?

API Layers

API Gateway The entry point for all incoming requests. Responsible for authenticating the request, rate limiting, and routing traffic to the correct downstream service.
BFF Layer An abstraction layer primarily used for public APIs or when serving multiple platforms (e.g. mobile and web). It decouples the frontend contract from the backend schema — so if the database schema changes, only the BFF layer needs updating, leaving the frontend untouched.
Validation Ensures incoming data is correct and meets required constraints before any processing occurs.
DTO Data Transfer Objects shape and transform data as it moves between layers, keeping concerns cleanly separated.
Service Layer Contains the core business logic of the application.
Model / Repository Handles communication with the database, abstracting away the underlying queries or ORM details.

Why Separate?

Separation of concerns — each layer has a single responsibility, making the system easier to test, maintain, and scale independently. If one layer changes, the others remain unaffected.

Q — 02

Tell us about a technical decision you made that had a significant impact on a project. How would you explain your reasoning to a client?

Context

On a web application project, I made the decision to introduce a BFF (Backend for Frontend) layer to serve both the web and mobile clients from a single, unified backend. Mobile was becoming a requirement, and without this layer, we would have needed a separate API service to handle the different data shapes each platform needed — meaning duplicated logic, more services to maintain, and higher long-term cost.

The Tradeoff

Building the BFF layer added development time upfront. But the long-term benefits outweighed that cost — mobile could be supported without spinning up an entirely new API service, keeping the system clean and maintenance cost low.

Explaining to a Client

// Client-friendly explanation

"Think of the BFF as a translator sitting between your app and your users. Instead of building a separate backend every time you add a new platform — like mobile — this layer handles that for you. It cost us some extra time upfront, but it means your mobile app and web app can grow independently without duplicating infrastructure. Long term, it saves time, reduces risk, and makes the system easier to scale."

Q — 03

Describe a time when you had to fix a complex system to improve its scalability or performance. What was the initial issue, what changes did you make, and what were the results?

The Problem — EzyVet

While working at EzyVet on a mobile application, the app was fetching data from three third-party sub-company APIs and aggregating the results on the client side. As usage grew, we started hitting the rate limits of those external APIs — causing slow response times and in some cases showing users an error page.

The core issue was that the app was constantly polling the external APIs for data. The more users we had, the more requests we fired, and the faster we burned through rate limits. It didn't scale.

The Decision

Instead of continuing to pull data from the APIs, I proposed flipping the model entirely. Rather than us fetching from them, we'd have the third-party services broadcast data to us via events (webhooks). We'd then store that data in a local database within our system. The app would query our own database instead of hitting external APIs directly.

Results

↑ Response time improved ✓ Rate limit errors eliminated ↑ Scales with user growth ✓ No extra API services needed

Explaining to Stakeholders

// Stakeholder-friendly explanation

"Previously, every time a user opened the app, we were knocking on three different doors asking for data — and those doors had limits on how many times we could knock. Instead, we asked those services to notify us whenever something changed, and we stored that information ourselves. Now when a user opens the app, we answer from our own records instantly. It's faster, more reliable, and won't break as we grow."
